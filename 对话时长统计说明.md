# 对话时长统计说明（工时 / `activeMs`）

> 本文说明 shine-code-submit 中**「对话总时长」**（报表/KPI 里的「工时」，字段 `activeMs`）的统计逻辑。
> 代码主入口：`src/daemon/transcript.ts` 的 `sessionActiveMs`。
>
> 结论先行：**它是估算值，不是真实计时**。度量的是「与 Claude 交互的活跃度」，不是「你工作了多少小时」。

---

## 1. 它是什么

每个 Claude Code session 算一个 `activeMs`（毫秒），代表该会话「与 Claude 交互的活跃时长」。随 report 上报到 tokenserver 报表站显示（KpiCards + 会话表时长列）。

⚠️ **不要当成人工工时**。它看不见「离开 Claude 自己干活」的时间。

## 2. 为什么是估算（而非真实计时）

Claude Code 的 transcript jsonl（`~/.claude/projects/<project>/<session>.jsonl`）**不落盘 duration** —— `cost.total_duration_ms` 是运行时字段，进程结束就没了。磁盘上只有每条 message 的**时间戳**。所以只能用 message 时间戳序列去**估算**活跃时长。

## 3. 核心算法 `sessionActiveMs`

位置：`src/daemon/transcript.ts:473`

### 步骤

**① 收集 usage 条目**
- 读父 transcript + 同目录 `subagents/*.jsonl`（`sessionTranscriptFiles`）
- 按 `usage` 字段存在性过滤：`if (!line.includes('"usage":{')) continue`（`transcript.ts:359`，代码不判 role；实际只有 assistant 消息携带 usage）
- 过 ccusage 全部门校验：`hasUnsupportedNullField`（null 黑名单）/ `isValidCcusageTimestamp`（严格 ISO8601）/ `isValidUsageEntry`（semver + 非空字段）
- 每条记 `ts`（时间戳）+ token 四字段 + `messageId`/`requestId`/`isSidechain`/`hasSpeed`

**② 去重（与 token 完全同源）**
- `dedupedSurvivors` → `pushDedupedEntry`
- 按 `messageId + requestId` 精确键去重，sidechain 重放兜底
- 偏好：非 sidechain → total 更大 → 带 speed
- ⚠️ **必须走 messageId 去重，不能用 `Set(ts)`** —— 同 messageId 的多行可能 ts 不同（sidechain 重放/重试），Set 去不掉，会重复计时

**③ 收集时间戳**：`survivors.map(e => e.ts)`，filter `Number.isFinite`，升序排序

**④ gap-aware 切 burst 累加**：

```
GAP_MS    = 3_600_000   // 1h：相邻 ts 间隔 > 1h 视为用户离开，断 burst
BUFFER_MS =   600_000   // 10min：每段 burst 末尾补的读/测时间

active = 0
burstStart = ts[0], prev = ts[0]
for i in 1..n-1:
  if ts[i] - prev > GAP_MS:        // gap，前段 burst 结算
    active += (prev - burstStart) + BUFFER_MS
    burstStart = ts[i]
  prev = ts[i]
active += (prev - burstStart) + BUFFER_MS   // 末尾 burst
```

## 4. 关键常量（`transcript.ts:457-458`）

| 常量 | 值 | 含义 |
|---|---|---|
| `GAP_MS` | 1h | 相邻 timestamp 间隔超过此值 → 视为用户离开，切新 burst |
| `BUFFER_MS` | 10min | 每段 burst 末尾补的「读回复 / 思考 / 测试」时间 |

## 5. 缓存与数据流

```
transcript.ts  sessionActiveMs          ← 真正算法
    ↑
token-cache.ts getSessionActiveMs       ← 复合 mtime 缓存（父 + subagents 所有文件 mtime 拼成 key）
    ↑                                     文件没变直接返回，变了才重算
claude-scan.ts scanSessions             ← 遍历所有 transcript，每 session 取 activeMs（2s TTL）
    ↑                                     0-token 空 transcript 在此被跳过，不进列表
server.ts     buildReport (line 354)    ← session 级透传 activeMs 进 report
    ↓
POST reportUrl → tokenserver            ← 上报到报表站
    ↓
tokenserver 前端                         ← KpiCards / 时长列展示
```

**为什么两层缓存**：`/api/sessions` 每 2s 被前端轮询，逐 session 重读 transcript 汇总会很重。`getSessionActiveMs` 用复合 mtime key（父 + 所有 subagents 任一文件没变就命中），`scanSessions` 自己再叠 2s TTL。

## 6. 边界情况

| 情况 | 结果 | 说明 |
|---|---|---|
| 0 条 usage（空 session） | `0` | 且在 `claude-scan.ts:136` 被 0-token 跳过，**不进 report**，报表看不到 |
| 1 条 usage（单点 burst） | `10min` | `(T-T) + BUFFER_MS`，buffer 兜底，避免「只发一条=0」 |
| N 条在同 burst 内 | `实际跨度 + 10min` | 例：ts 跨 12min → 22min |
| 多段 burst（gap 断开） | `Σ(每段跨度 + 10min)` | 每段独立加 buffer |

> **「最少 10 分钟」的含义**：每段 burst 末尾无条件 `+ BUFFER_MS`，单点 burst 时实际跨度为 0，得 10min。所以任何进报表的 session，`activeMs` 数学下限就是 10min —— 这是算法输出值的下限，**不是真实至少花了 10 分钟**。一句 30 秒的问答也算 10min。

## 7. 计算示例

### 例 1：一问一答（无工具）
`ts = [09:00]`（1 条）→ 单点 burst：`0 + 10min` = **10min**（不管真实看了多久）

### 例 2：连续多轮（间隔都 < 1h）
`ts = [10:00, 10:05, 10:12]`（一个 burst）→ 跨度 `12min + 10min` = **22min**

### 例 3：9 点问 → 自己写代码 3 小时 → 12 点检查
`ts = [09:00, 12:00]`（2 条，gap = 3h）
```
i=1: 12:00 - 09:00 = 3h > 1h → 结算前段
     burst1 = (09:00-09:00) + 10min = 10min
末尾: burst2 = (12:00-12:00) + 10min = 10min
合计 = 20min
```
→ **20min**（中间自己写代码的 3 小时**完全不计**，被 1h gap 截断丢弃）

## 8. 聚合现状 ⚠️

- **session 级**：有 `activeMs`
- **project 级**：`buildReport` 的 project 对象只有 `totalTokens` + `totalLines`，**没有 `totalActiveMs`**
- **全局**：后端不提供

总工时 / 每日时长由 **tokenserver 前端 `derive.ts` 现场从 session 级 `activeMs` 聚合**（后端 report 不提供）：`globalTotals().activeMs`（`derive.ts:265`，从 `flattenSessions` 累加）、`dailyStats().dur`（`derive.ts:243`，按日）。`derive.ts:264` 注释明确「不给 UserAgg 加 totalActiveMs」，故天然适配时间范围过滤（`filterUsersByRange` 后重算仍准）。`fmtDuration`（`derive.ts:122`）格式化。

## 9. 展示位置

- **本地 dashboard**（livesetting `ui/`）：**不显示**工时。`ui/` 源码无 `activeMs` / `工时` / `formatDuration`
- **tokenserver 报表站**（47.98.221.20）：显示。KpiCards「对话总时长」卡（`fmtDuration(globalTotals.activeMs)`，`KpiCards.tsx:36-38`）+ 每日时长 sparkline（`dailyStats.dur`）+ 会话表时长列（`RecentSessionsTable`）

## 10. 本质与局限

这个指标度量的是**「与 Claude 交互的活跃度」**，不是**「你工作了多少小时」**：

| 场景 | 是否计入 | 说明 |
|---|---|---|
| 离开 Claude 自己干活（写代码 / git / 查资料） | ❌ 不计 | transcript 天生拿不到 editor/git 活动 |
| 中间长时间不用 Claude | ❌ 不计 | 被 1h `GAP_MS` 截断丢弃 |
| 快速一问一答 | ⚠️ 固定 10min | 可能**高估**（真实可能 30 秒） |
| 长时间连续交互但消息时间戳密集 | ⚠️ 只算跨度 | 可能**低估**（真实投入 > ts 跨度） |

- **适合**：反映「Claude 被用了多少 / 交互密度」
- **不适合**：当真实人工工时报出去

## 12. 调参与改进方向

- **改阈值**：调 `GAP_MS` / `BUFFER_MS`（`transcript.ts:457-458`）。`BUFFER_MS` 调大 → 短交互高估更严重；`GAP_MS` 调大 → 更宽容地把分散交互算连续。
- **加 project / 全局聚合**：在 `buildReport` 的 project 对象加 `totalActiveMs`（仿 `sumTokens` 对 `activeMs` 求和）。
- **本地 dashboard 显示工时**：在 `ui/` 加 `activeMs` 展示（目前只有 tokenserver 显示）。
- **更准的工时**：需要额外数据源（editor 活动时间 / git commit 时间），transcript 天生拿不到「用户在不在」。
