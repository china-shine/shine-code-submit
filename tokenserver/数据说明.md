# AI 效能平台 数据说明

> 本文档逐项说明 tokenserver(AI 效能平台)dashboard 上展示的**每一项数据**:它是什么(含义)、怎么算出来的(计算规则)、对应后端哪段代码。
>
> 阅读建议:先看「二、通用口径」,后面每项数据都引用这些口径。

---

## 一、数据从哪来

```
Claude Code transcript(.jsonl)
   │  shine-code-submit daemon 扫描(逐行解析/校验/去重,口径对齐 ccusage)
   ▼
daemon 每分钟上报 → POST /api/report(gzip)
   │  tokenserver saveReport 落库
   ▼
tokenserver SQLite:sessions / projects 两表
   │  前端请求时二次聚合
   ▼
Dashboard(/api/stats · /api/sessions · /api/member)
```

- **sessions 表**(每行 = 一个会话):`sessionId`(主键)、`gitUser`、`cwd`、`lastActive`(最后活跃,毫秒时间戳)、`input`/`output`/`cacheCreation`/`cacheRead`(token 四字段)、`added`/`deleted`/`modified`(代码行三字段)、`activeMs`(活跃时长)、`title`、`updatedAt`。
- **projects 表**(每行 = 一个用户×项目):主键 `(gitUser, cwd)`,`name`、`gitRemote`、`lastActive`。
- **去重**:同一 `sessionId` 重复上报时,仅当新数据 `lastActive ≥ 旧` 或 `cwd` 变了才覆盖(幂等)。

三个聚合接口(路由见 `src/server.ts`):
| 接口 | 作用 | 后端函数 |
|---|---|---|
| `GET /api/stats` | 总览全部聚合 + 成员列表 | `getStats` |
| `GET /api/sessions` | 会话明细分页(总览表 / 成员详情表) | `getSessions` |
| `GET /api/member/:gitUser` | 单成员 KPI + 趋势 | `getMember` |

## 二、通用口径(先读这节)

### 1. Token 四字段 与 rawTotal

每个会话有四个 token 字段(来自 transcript `message.usage`,口径与 [ccusage](https://github.com/ccusage/ccusage) 一致):

| 字段 | 含义 |
|---|---|
| `input` | 输入 token |
| `output` | 输出 token |
| `cacheCreation` | 缓存写入(prompt caching 创建) |
| `cacheRead` | 缓存读取 |

- **rawTotal = input + output + cacheCreation + cacheRead**(四字段全量)。**凡页面上写「总 Token」的,都是 rawTotal**。
- **缓存口径 cache = cacheCreation + cacheRead**(趋势图「缓存」视图、饼图「缓存 Token」)。
- **inoutTokens = input + output(不含缓存)** —— 只用作「Token 效率(行/M)」的分母,见下。

### 2. 代码行 lines

`added`(新增) / `deleted`(删除) / `modified`(修改) 三字段,来自 daemon 上报。

- **lineTotal = added + deleted + modified**。凡「代码行数 / 生成代码行数」都是三者之和。
- 会话表「代码变更」列**分别**显示 `+added -deleted Mmodified`,不合并。

### 3. 对话时长 activeMs

- daemon 上报的 **gap-aware 活跃时长**:把会话内 ≥ 1 小时的间隙截断后再累加,**不是墙钟跨度**(自己发呆的时间不计)。
- 所有「对话时长」= 范围内各会话 `activeMs` 直接求和,格式 `0m`(为 0)/ `<1m` / `Xm` / `Xh`(整点)/ `Xh Ym`。
- 老版本 daemon 可能缺失该字段(显示 0)。

### 4. 日期范围(闭区间,按 lastActive)

前端传 `start` / `end`(YYYY-MM-DD),后端 `parseDateRange`(`src/server.ts`):

- **from** = 开始日 `00:00:00.000`(本地时区);留空 = `0`(不限起始)。
- **to** = 结束日 `23:59:59.999`(即次日 0 点 − 1ms);留空 = `Number.MAX_SAFE_INTEGER`(不限结束)。
- 过滤条件 `WHERE lastActive >= from AND lastActive <= to`,**闭区间**,会话按**最后活跃时间**归属(不是开始时间)。

> 「重置」按钮:开始 = 全量最早记录(`dataMin`)、结束 = 全量最新记录(`dataMax`),即范围 = 全部数据。
> 「开始日期 ≤ 结束日期」:日历里晚于结束日的、早于开始日的日期会被禁选。

### 5. 粒度 granularity(只影响趋势图)

趋势图可按 `day` / `week` / `month` 聚合(`bucketOf`,`src/store.ts`):

- **day**:日历日,标签 `MM-DD`
- **week**:周一起算的自然周
- **month**:自然月,标签 `YYYY-MM`

> 当前 UI 固定按「日」聚合。KPI 卡的 sparkline 始终按「日」,不受粒度影响。

### 6. 成员 / 项目过滤

- **成员筛选**:URL `members=用户A,用户B`,SQL `gitUser IN (...)`,影响聚合数据(stats)和会话表(/api/sessions)。
- **`allMembers`**(成员下拉选项)= 全表 `SELECT DISTINCT gitUser`,**不受任何过滤影响**。
- **`activeMin/activeMax`** = 当前过滤后数据的 lastActive 范围(顶部「范围徽章」用)。
- **`dataMin/dataMax`** = 全表(无过滤)的 lastActive 范围(重置按钮用)。

### 7. 「真项目」判定(`isRealProjectCwd`)

项目榜、活跃项目数只统计「真项目」,排除盘根/家目录/桌面等。按 cwd 原值(统一 `/`)拆段判定:

| 情况 | 判定 |
|---|---|
| 空 / 段数 ≤ 1(如 `C:`) | ❌ 非真项目 |
| 段数 3 且第 2 段 = `users`(如 `C:/Users/xxx`) | ❌ 家目录 |
| 段数 4 且 = `users/.../{desktop,documents,downloads}` | ❌ 桌面/文档/下载 |
| 其他 | ✅ 真项目 |

> token 总量、对话次数、代码行、成员榜等**不依赖**真项目判定,用全部会话。

### 8. ⚠️ 0-token 会话的口径差异(务必注意)

有些会话四字段 token 全为 0(如纯读取、出错)。它们在不同地方口径不同:

| 是否计入 0-token 会话 | 涉及数据 |
|---|---|
| **计入**(不过滤) | KPI「对话次数」、**KPI「活跃项目」**、对话时长、代码行、总 Token、趋势、饼图、成员榜、**项目榜**、成员「对话次数」 |
| **不计入**(过滤 `token>0`) | 会话规模分布、最近会话表(总览 / 成员详情)的 `total` 与列表行 |

**因此**:KPI「对话次数」 ≥ 最近会话表的总条数(当存在 0-token 会话时)。这是正常的,不是 bug。

## 三、总览页

### 3.0 页面顶部状态

- **标题**:「数据总览」/「成员分析」(按所在页)。
- **状态行**:`{活跃成员} 成员 · {会话数} 会话 · 更新于 {刷新时刻}` —— 成员数 = `totals.members`、会话数 = `totals.sessions`(含 0-token),刷新时刻 = 本次拉取时间。
- **范围徽章**(日期框右侧):`{activeMin} — {activeMax}`,当前过滤后数据的**实际活跃时间范围**;无数据显示「无数据」。注意:这是数据实际范围,不一定等于你选的日期范围(数据未必覆盖那么全)。

### 3.1 KPI 卡(6 张)

数据来自 `stats.totals`(全局聚合)+ `stats.daily`(按日,sparkline 用)。

| KPI | 含义 | 计算规则 |
|---|---|---|
| **总 Token 消耗** | 范围内所有会话的原始 token 总量 | `rawTotal = Σ(input+output+cacheCreation+cacheRead)`;副标拆「输入 / 输出」;sparkline 按日 total |
| **对话次数** | 范围内会话条数(每条 = 一个 sessionId) | `Σ 会话行数`(含 0-token 会话);sparkline 按日会话数 |
| **对话总时长** | 各会话 gap-aware 活跃时长之和 | `Σ activeMs`;副标「gap-aware 估算 · 1h 间隙截断」 |
| **生成代码行数** | added+deleted+modified 之和 | `Σ(added+deleted+modified)`;副标拆「新增/修改/删除」 |
| **活跃成员** | 范围内出现过的 gitUser 去重数 | `memberAcc.size`(无 sparkline) |
| **活跃项目** | 范围内「真项目」数,**跨用户计数** | `realProjKeys.size`,key = `gitUser\0cwd`(同 cwd 不同用户算两个) |

> sparkline 数据来自 `stats.daily`(固定按日),不足 2 个点则不画。

### 3.2 Token 消耗趋势图

数据 `stats.trend`(按当前粒度分桶,**UI 固定为日**)。4 个视图切换:

| 视图 | 计算 |
|---|---|
| **总量** | `input + output + cacheCreation + cacheRead` |
| **输入** | `input` |
| **输出** | `output` |
| **缓存** | `cacheCreation + cacheRead` |

会话按 `lastActive` 归入对应日/周/月桶,桶内求和。顶部显示「首桶 – 末桶」日期范围。Y 轴用 `fmtK`(k/M/B),Tooltip 显示精确值。

### 3.3 Token 构成饼图

数据 `stats.composition = { input, output, cache }`,中心总数 = 三段之和(= rawTotal)。

| 段 | 计算 |
|---|---|
| 输入 Token | `Σ input` |
| 输出 Token | `Σ output` |
| 缓存 Token | `Σ(cacheCreation + cacheRead)` |

占比 = 段值 / 中心总数 × 100%(保留 1 位小数)。

### 3.4 Token 消耗排行(成员 / 项目 两 tab)

**成员榜**:`{ gitUser, token }`,`token` = 该成员全部会话的 **rawTotal**(含缓存),按 `token` 降序,取前 6。点行跳成员详情。

**项目榜**:`{ cwd, name, token }`,**仅累计「真项目」**会话,按 `(gitUser, cwd)` 聚合(同 cwd 不同用户会拆成两行),`token` = 该项目会话 rawTotal 之和,降序前 6。项目名取 projects 表 `name`;**name 为空时后端预填完整 cwd**(不是末段)——前端 `displayProjectName` 仅当 name 为纯数字/单字符时才回退到 cwd 末两段。

### 3.5 代码产出排行

`{ gitUser, lines, convs, token }`,按 `lines`(代码行)降序前 6。三列:

| 列 | 计算 |
|---|---|
| 代码行 | `Σ(added + deleted + modified)` |
| 次对话 | 该成员会话数(含 0-token) |
| **效率** | `代码行 / ((input+output) Token / 1,000,000)` = **行 / M Token**(input+output 不含缓存);分母为 0 显示 0 |

### 3.6 会话规模分布

把每个会话按其 rawTotal 规模归桶(**仅 token>0 的会话计入**):

| 桶 | 范围 |
|---|---|
| 0–10K | ≤ 10,000 |
| 10–100K | ≤ 100,000 |
| 100K–1M | ≤ 1,000,000 |
| 1–10M | ≤ 10,000,000 |
| >10M | 更大 |

柱状图,Tooltip 显示「N 个」。

### 3.7 最近会话表

`/api/sessions` 分页(默认每页 20)。**过滤 token>0**,按 `lastActive` 降序。列:

| 列 | 字段 / 计算 |
|---|---|
| 最后活跃 | `lastActive`(`YYYY-MM-DD HH:MM`) |
| 成员 | `gitUser`(空显示「未知」)+ 头像 |
| 标题 | `title`,空则 `sessionId` 前 8 位 |
| 项目 | `name` 优先(projects 表);空则完整 cwd;仅纯数字/单字符名才取 cwd 末两段(`displayProjectName`) |
| 输入 | `input` |
| 输出 | `output` |
| 总 Token | `input+output+cacheCreation+cacheRead`(rawTotal) |
| 时长 | `activeMs` |
| 代码变更 | `+added -deleted Mmodified` |

> 这里的「总条数」可能 < KPI「对话次数」,因为 0-token 会话被过滤掉了。

## 四、成员分析页

### 4.1 成员列表

数据 `stats.members`(`MemberAgg[]`),默认按 rawTotal 降序。列:

| 列 | 计算 |
|---|---|
| 成员 | `gitUser` |
| 最后同步 | 该成员会话 `max(lastActive)` |
| 活跃项目 | 该成员「真项目」cwd 去重数 |
| 对话次数 | 该成员会话数(含 0-token) |
| 对话时长 | `Σ activeMs` |
| 总 Token | `rawTotal`(四字段含缓存) |
| 代码行 | `Σ(added+deleted+modified)` |
| 效率(行/M) | `代码行 / ((input+output)/1M)` |
| 操作 | 「详情」/ 点行 → 进成员详情 |

> 成员列表「活跃项目」按 cwd 去重(单成员视角);总览 KPI「活跃项目」按 `gitUser\0cwd` 跨用户计数。对单成员而言两者等价,跨成员汇总时不同。

### 4.2 成员详情

数据:`/api/member/:gitUser`(个人 KPI + 趋势)+ `/api/sessions?member=`(个人会话表)+ 全局 `stats.totals`(团队均值)。

**头部**:头像 + 成员名(`gitUser`)+ 「最后同步 {该成员会话 max(lastActive)}」。

**6 个 KPI**:

| KPI | 计算 |
|---|---|
| 对话次数 | 该成员会话数(含 0-token) |
| 对话总时长 | `Σ activeMs` |
| 总 Token | rawTotal |
| 代码行数 | `Σ(added+deleted+modified)` |
| 活跃项目 | 该成员「真项目」cwd 去重数 |
| Token 效率 | `代码行 / ((input+output)/1M)` 行/M |

**Token 使用趋势**:按日聚合的 rawTotal,单条「总量」线(Y 轴 fmtK)。

**与团队均值对比**:团队人数 ≤ 1 时不显示。三项对比,personal = 本人数值,avg = 全局 totals ÷ 全局活跃成员数(**含本人**):

| 项 | personal | 团队均值 |
|---|---|---|
| Token 消耗 | 本人 rawTotal | `全局 rawTotal / 全局成员数` |
| 对话次数 | 本人会话数 | `全局会话数 / 全局成员数` |
| 代码行数 | 本人代码行 | `全局代码行 / 全局成员数` |

> 「团队均值含本人」:分母是范围筛选后的全部活跃成员(包括正在看的人),所以严格说是「全员平均」而非「其他人平均」。

**该成员最近会话表**:复用总览的会话表组件,规则相同(过滤 token>0、按 lastActive 降序)。

## 五、附录

### 5.1 容易误解的点

1. **「对话次数」vs 会话表条数**:前者含 0-token 会话,后者不含 → 前者 ≥ 后者。
2. **「活跃项目」两种口径**:总览跨用户计数、成员列表/详情单成员 cwd 去重。
3. **Token 效率分母用 input+output(不含缓存)**:缓存命中会拉高 rawTotal 但不增加「有效」token,效率用 inout 更合理。
4. **对话时长非真实工时**:gap-aware 估算,1h 间隙截断;自己写代码的间隙不计、多会话并行不叠加去重。
5. **团队均值含本人**:不是「除自己外的平均」。

### 5.3 上游口径

token 四字段、去重、session 归并的**上游算法**在 shine-code-submit daemon 侧(`src/daemon/transcript.ts` 等),与 ccusage `claude session` 逐字段对齐。tokenserver 只做存储 + 二次聚合,不改口径。详见仓库根 `README.md`「Token 统计逻辑」章节。
