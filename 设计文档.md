# Shine Code Submit 设计文档

## 1. 目标

利用 Claude Code 的 Hook 生态，结合一个本地常驻守护进程（Daemon），实现**复杂逻辑、状态管理与持久化**，且不拖慢 Claude Code 主进程响应。

> 核心思想：Hook 脚本只做「采集 + 落盘 + 转发」并立即退出，重活交给后台 Daemon 异步处理。
> 可靠性原则：**异步 ≠ 可丢，但允许重放**。Hook 落盘后再转发；Daemon 崩溃要自愈；事件不丢、处理幂等。

---

## 2. 架构

```
┌───────────────┐  事件(轻量)     ┌──────────────────────┐
│  Claude Code  │ ───────────────▶ │  Hook 进程(短命)        │
│ (生命周期事件) │                  │  1. 采集 env + stdin    │
└───────────────┘                  │  2. 写 spool 文件(原子)  │
                                   │  3. POST(失败才探测/拉起) │
                                   └────┬───────────────┬───┘
                         健康路径 POST │               │ spool 目录
                        (单次往返)    │               │ (每事件一文件)
                                       ▼               ▼ (回捞)
                         ┌──────────────────────────────┐
                         │  本地 Daemon(常驻)               │
                         │  · 鉴权 HTTP API(事件/统计/健康)   │
                         │  · spool 消费:读→幂等入库→unlink  │
                         │  · SQLite(按 cwd 隔离) · 异步重算 │
                         │  · 日志 + 积压指标                │
                         │  · 查看页(只读 Dashboard)         │◀───┐
                         └──────────────────────────────┘    │
                              ▲                              │
                              │ HTTP/WS(仅本机)                │
                              │                               │
                         ┌────┴─────────────┐                 │
                         │  浏览器 Dashboard  │  shine-code-submit ui │
                         │  (实时事件/指标)   │  ──────────────┘
                         └──────────────────┘
```

两大部件 + 一个查看入口：

| 部件 | 角色 | 技术选型 |
| --- | --- | --- |
| **Daemon** | 常驻后台进程，持有状态、对外 API、执行耗时任务 | Bun（`Bun.serve`、`bun:sqlite`） |
| **Hook** | 被 Claude 唤醒的轻量入口，落盘 + 转发后退出 | `bun build --compile` 产出的单文件二进制（避免每次起 runtime 的冷启动成本） |
| **查看页** | 本机浏览器查看运行状态与事件流 | Daemon 内置静态页 + 原生 JS，无前端框架 |

## 3. 数据流 / 生命周期

Daemon 监听 `127.0.0.1:36666`。

### Hook 侧（每次事件，进程内顺序执行）

1. **采集**：读 `env` + `stdin`（Claude 注入的 JSON），补 `cwd`、`sessionId`、`pid`、`event_id`。
2. **落盘 spool**（唯一必成功环节）：写一个独立文件到 spool 目录，**先写 `.tmp` 再 `rename`**（NTFS/POSIX 同分区 rename 原子）。文件名 `{ts}-{pid}-{rand}` 天然唯一，无并发冲突。
3. **热转发**：`fetch POST` 到 Daemon，超短超时（~500ms）。

**故障路径**（仅在 POST 连接失败时触发，健康路径不进入）：
`ECONNREFUSED` → `GET /api/health` 探测 → 校验是「自己人」→ 不存在则 detached 拉起 → 轮询 health 至 ready → 重读 pid 文件拿 token → 重试 POST。

> 关键：**热路径默认单次往返**；探测/拉起只在故障时发生。无论如何事件已在 spool，最终一致。

### Daemon 侧双消费

- **热路径**：HTTP 收到事件 → 幂等入库 → 异步处理。
- **回捞路径**：后台扫 spool 目录 → 读文件 → 幂等入库 → 处理 → `unlink`（删除即确认）。

两条路径共享同一套幂等逻辑，互为兜底：热路径宕了靠回捞，回捞慢了靠热路径。

## 4. 关键设计约束

### 4.1 可靠性（核心）

- **目录式 spool，每事件一文件**：规避多进程并发 append 同一文件导致的交错/截断（Windows append 非原子）。
- **幂等 = 恰好一次的外在表现**：每事件带稳定 `event_id`；入库以 `(sessionId, event_id)` 唯一约束 `INSERT OR IGNORE`；业务处理假设可能重放。
- **落盘失败的兜底**：spool 写失败（磁盘满 / 权限错 / 父目录缺失）时，向 stderr 写一行含事件摘要的告警（Claude 会记录 hook stderr），仍退出码 0 不崩主进程。即"极端情况下可能丢，但**可被发现**"，而非静默消失。

### 4.2 不阻塞主进程

- Hook 热转发超短超时、失败静默吞掉，绝不抛错影响 Claude Code。
- 重计算（总结、入库、代码分析）一律丢给 Daemon 异步处理。
- **不预先做 health 探测**——直接 POST，失败才走故障路径，避免健康路径下的多余往返。

### 4.3 自启动与自愈

- `SessionStart` 是首次拉起的常规触发点；但**任意一次事件的故障路径都能触发复活**（POST 失败 → 探测 → 拉起）。
- 拉起后**轮询 health 至 ready 或超时**；超时则放弃热转发，事件靠 spool 回捞。
- **认出自己人**：`/api/health` 返回 `{"service":"shine-code-submit","version","pid","uptime"}`；Hook 必须校验 `service=="shine-code-submit"` 才认端口归属，避免 `36666` 被无关程序占用时误判。

### 4.4 安全

- Daemon **只绑定 `127.0.0.1`**，禁止 `0.0.0.0`。
- **鉴权**：`/api/health` 不鉴权（仅状态）；其余端点（含查看页的数据接口）要求 `Authorization: Bearer <token>`。
- **token 与启动时序**：Daemon 启动时绑定端口 → 生成随机 token → 写 pid 文件（`pid`、`port`、`token`、`startedAt`，仅当前用户可读）。Hook / 查看页均从 pid 文件取 token。
  - **化解循环依赖**：首个事件若发生在 Daemon 起来之前，POST 会因连接失败走故障路径；即便鉴权暂时缺失，事件已在 spool，回捞时 Daemon 已 ready、token 已就位 → 最终成功。无需"拉起期特殊免鉴权"。

### 4.5 状态隔离

- 每事件带 `cwd`（`process.cwd()`）与 `sessionId`。
- spool 文件内容含 `cwd`/`sessionId`；SQLite 按 `cwd` 分库或分表，避免不同项目/终端数据串位。

### 4.6 平台：Windows（当前环境）

- `spawn` 拉起 Daemon：`detached: true`、`windowsHide: true`、`stdio` 重定向到日志文件（见 4.7），否则黑窗或随父进程退出。
- 路径用 `%LOCALAPPDATA%/shine-code-submit/`（spool、pid、log 子目录）。
- Bun 与 Node 在 Win 上 spawn 语义有差异，拉起/轮询需实测。

### 4.7 进程生命周期与可观测性

- pid 文件记录 `pid`、`port`、`token`、`startedAt`；启动时若端口被占且 health 校验通过则复用，不重复拉起。
- 提供 `shine-code-submit stop | restart | ui` 管理命令，避免孤儿/僵尸进程。
- **可观测性**（detached 进程否则成黑盒）：
  - 日志：Daemon stdout/stderr 重定向到 `daemon.log`，按天轮转。
  - `GET /api/stats`：spool 积压数、处理速率、最近错误、uptime。
  - 落盘失败、鉴权失败、回捞异常均写日志，便于排查。

## 5. Hook 事件类型（按需注册）

Claude Code 共 9 个 hook 事件；本插件注册 7 个**只读观测**事件（hook 退出码恒 0，不影响主进程）。

**已注册（7）：**
- `SessionStart` — 会话开始 / resume / clear / compact（兼做 Daemon 首次拉起点）
- `UserPromptSubmit` — 用户提交提示词前
- `PostToolUse` — 工具调用完成后
- `Stop` — 主 agent 结束响应
- `SubagentStop` — 子 agent（Task 工具）结束响应
- `PreCompact` — 上下文压缩前（手动 `/compact` 或自动）
- `SessionEnd` — 会话结束（触发该 session 内存状态淘汰，淘汰策略随业务定）

**未注册（2）：**
- `PreToolUse` — 工具调用前。**注意：其退出码/返回 JSON 会阻断或改写工具调用，与「Hook 不影响主进程」前提冲突。** 仅当业务确需拦截时启用，且不能套用"落盘即退出"模板，需单独设计同步返回逻辑。
- `Notification` — 权限请求 / 闲置通知；噪音大、观测价值低，默认不收。

> hook 侧 `VALID_TYPES`（`src/hook/main.ts`）与 `HookEventType`（`src/shared/types.ts`）为事件名白名单，新增事件需同步这两处；daemon 侧 `/api/hook/:type` 不做白名单，按 URL 原样入库。

## 6. 接口契约

| 端点 | 方法 | 鉴权 | 用途 |
| --- | --- | --- | --- |
| `/api/health` | GET | 无 | 存活 + 归属确认（Hook 探测/拉起判断用） |
| `/api/hook/<event>` | POST | token | 接收事件（热路径） |
| `/api/stats` | GET | token | 积压/速率/错误/uptime 等运行指标 |
| `/api/events` | GET | token | 历史事件查询，支持 `?cwd=&sessionId=&limit=&since=` 筛选 |
| `/api/ws` | WS | token | 实时事件推送 + 指标更新（查看页与未来面板共用） |
| `/ui` | GET | 见下 | 返回查看页静态资源 |

## 7. 查看页面（Dashboard）

**定位**：本机浏览器打开的只读监控页，把 §4.7 的指标与事件流可视化出来，方便排查与观察。v1 只读，不提供任何写/控制操作，避免误操作影响 Daemon。

### 访问与鉴权

- 命令 `shine-code-submit ui` 读 pid 文件取 token，调用系统默认浏览器打开 `http://127.0.0.1:36666/ui?t=<token>`。
- `/ui` 的**静态资源本身放开**（HTML/JS 无敏感数据）；所有数据接口（`/api/stats`、`/api/events`、`/api/ws`）仍需 token。
- 前端从 URL 取 `t` 存入 `sessionStorage`，随即 `history.replaceState` 清掉 URL 中的 token，关闭标签页即失效。
- 仅本机访问（绑 `127.0.0.1`），不对外。

### 页面内容

- **顶部状态条**：service/version、uptime、spool 积压、处理速率、最近错误时间（轮询 `/api/stats`）。
- **左侧 Session 树**：按 `cwd` 分组列出 session，显示最后活跃时间与状态；点击即筛选主区。
- **主区事件流**：`/api/ws` 实时推送 + 进入时拉 `/api/events` 补齐历史；按选中 session 筛选，每条显示时间、事件类型、`toolName`、摘要。
- **底部日志尾**：`/api/stats` 附带最近 N 行 `daemon.log`，便于不切终端即可看现场。

### 形态

Daemon 内置单页静态资源（原生 HTML/JS + 少量 CSS，不引入前端框架），随 Daemon 一起分发，保持最简。

## 8. 实现里程碑

1. **Daemon 骨架**：`Bun.serve` 绑定 `127.0.0.1:36666`，实现 `/api/health`（含 service 标识）与事件接收端点（先打印日志）。
2. **目录式 spool + 回捞**：定义文件格式与 `event_id` 规则，Daemon 实现扫目录消费 + `unlink` 确认 + 定时回捞。
3. **幂等入库**：`(sessionId, event_id)` 唯一约束，热路径与回捞共用。
4. **Hook 二进制**：`bun build --compile` 产出入口，实现采集 → 原子落盘 → POST → 故障路径（探测/拉起/轮询/重试）。
5. **token 与 pid 文件**：启动写 pid 文件，端点鉴权，Hook 读取。
6. **可观测性**：日志重定向与轮转、`/api/stats`、失败告警落日志。
7. **查看页面**：`/ui` 静态页 + `/api/events` 查询 + `/api/ws` 推送；`shine-code-submit ui` 命令打开浏览器并带 token。
8. **管理命令**：`stop`/`restart`/`ui`，pid 复用判断。
9. **业务逻辑**：按第 9 节确定的方向接入 SQLite 与异步任务。
